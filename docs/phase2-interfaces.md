# Phase 2 — Interface Contracts

**Status:** Draft for review. SWE-Core and SWE-Ext implement to these types.
**Author:** TL agent, 2026-04-14.
**Parent:** [`phase2-passkey-design.md`](./phase2-passkey-design.md) §4, §7.

This file exists because Feature B crosses three packages and the integration seam has historically been where bugs hide. Fix the types here, discuss here, and then write code against them. Do not evolve these interfaces silently in PRs — propose a change, update this doc, link in the commit.

All types below assume TypeScript strict mode (no implicit any, no `@ts-ignore`). They match the 0-escape-hatch baseline for `packages/core`.

## 1. Core ↔ Extension protocol (Feature B runtime)

The extension background SW already talks to the core web app via an encrypted NaCl channel (see `packages/extension/src/background/protocol/protocol-impl.ts`). We add four verbs to that channel. They are transport-agnostic; the encryption + nonce layer is unchanged.

All types live in **`packages/extension/src/background/protocol/types.ts`** (extension side) and mirrored in **`packages/core/app/scripts/comp/extension/protocol-impl.ts`** (core side).

```ts
// --- passkey-create-entry ---

export interface KeeWebConnectPasskeyCreateEntryRequestPayload extends KeeWebConnectRequest {
    action: 'passkey-create-entry';
    rpId: string;                   // registrable domain, validated against origin
    rpName: string;                 // display name from options.publicKey.rp.name
    userHandle: string;             // base64url
    userName: string;               // display (not authoritative)
    userDisplayName: string;
    credentialId: string;           // base64url, 32 bytes, generated by SW
    alg: -7 | -8;                   // COSE algorithm
    privateKeyPkcs8: string;        // base64url, PKCS#8 DER
    publicKeySpki: string;          // base64url, SPKI DER
    transports: readonly ('internal' | 'usb' | 'nfc' | 'ble' | 'hybrid')[];
    targetGroupUuid?: string;       // if omitted, core picks default from settings
}

export interface KeeWebConnectPasskeyCreateEntryResponsePayload extends KeeWebConnectResponse {
    entryUuid: string;              // the KDBX entry that was created
    // on error, `error` + `errorCode` set per existing convention
}

// --- passkey-get-by-rp ---

export interface KeeWebConnectPasskeyGetByRpRequestPayload extends KeeWebConnectRequest {
    action: 'passkey-get-by-rp';
    rpId: string;
    // allowCredentials[]: if provided, limit to matching credentialIds
    allowCredentialIds?: readonly string[];
}

export interface KeeWebConnectPasskeyMatch {
    entryUuid: string;
    credentialId: string;           // base64url
    userHandle: string;             // base64url
    userName: string;
    userDisplayName: string;
    alg: -7 | -8;
    publicKeySpki: string;          // base64url
    signCount: number;              // uint32, current stored value
    lastUsedDate: string | null;    // ISO 8601 UTC
}

export interface KeeWebConnectPasskeyGetByRpResponsePayload extends KeeWebConnectResponse {
    matches: readonly KeeWebConnectPasskeyMatch[];
    // empty array = no stored passkey for this rpId
}

// --- passkey-sign ---

export interface KeeWebConnectPasskeySignRequestPayload extends KeeWebConnectRequest {
    action: 'passkey-sign';
    entryUuid: string;              // selected from passkey-get-by-rp matches
    authenticatorData: string;      // base64url; already includes incremented signCount
    clientDataHash: string;         // base64url; SHA-256 of clientDataJSON
    expectedSignCount: number;      // sanity check: must equal stored + 1
}

export interface KeeWebConnectPasskeySignResponsePayload extends KeeWebConnectResponse {
    signature: string;              // base64url, ASN.1 DER for ES256
    newSignCount: number;           // echoes persisted value after the +1
}

// --- passkey-update-counter (explicit) ---

// Optional convenience verb if we decide signCount persistence should be
// separated from the sign operation (e.g., for retry-safe protocols).
// Phase 2: probably not needed — passkey-sign persists atomically.
// Leaving the shape here in case B-3 integration work argues otherwise.

export interface KeeWebConnectPasskeyUpdateCounterRequestPayload extends KeeWebConnectRequest {
    action: 'passkey-update-counter';
    entryUuid: string;
    newSignCount: number;           // must be exactly storedSignCount + 1
}

export interface KeeWebConnectPasskeyUpdateCounterResponsePayload extends KeeWebConnectResponse {
    persisted: true;
}
```

**Error codes** (new):

| `errorCode` | Meaning | HTTP-ish mnemonic |
|---|---|---|
| `PASSKEY_RP_MISMATCH` | rpId not registrable-domain suffix of origin | 400 |
| `PASSKEY_VAULT_LOCKED` | no open vault at request time | 423 |
| `PASSKEY_NO_MATCH` | rpId/credentialId found nothing | 404 |
| `PASSKEY_USER_CANCELLED` | user declined the consent UI | 499 |
| `PASSKEY_COUNTER_REGRESSION` | expectedSignCount != stored+1 | 409 |
| `PASSKEY_UNSUPPORTED_ALG` | requested COSE alg not supported | 400 |
| `PASSKEY_WRITE_FAILED` | KDBX save failed (bubbles to user) | 500 |

The existing `KeeWebConnectResponse.errorCode` field string type is widened to `string` (it already is) — these are documented values.

## 2. KDBX entry schema for stored passkeys

The canonical list of `NKW.Passkey.*` field names — **do not add fields without updating this table and `NKW.Passkey.Version`**:

```ts
// Not executable code; documentation contract. Helper implementation lives in
// optional `packages/db/lib/helpers/passkey-entry.ts` (sub-issue B-6).

export interface NkwPasskeyEntrySchema {
    readonly 'NKW.Passkey.Version': '1';
    readonly 'NKW.Passkey.RpId': string;
    readonly 'NKW.Passkey.RpName': string;
    readonly 'NKW.Passkey.CredentialId': string;         // base64url
    readonly 'NKW.Passkey.UserHandle': string;           // PROTECTED, base64url
    readonly 'NKW.Passkey.UserName': string;
    readonly 'NKW.Passkey.UserDisplayName': string;
    readonly 'NKW.Passkey.Alg': '-7' | '-8';
    readonly 'NKW.Passkey.PrivateKey': string;           // PROTECTED, base64url PKCS#8
    readonly 'NKW.Passkey.PublicKey': string;            // base64url SPKI
    readonly 'NKW.Passkey.SignCount': string;            // decimal uint32
    readonly 'NKW.Passkey.CreatedDate': string;          // ISO 8601 UTC
    readonly 'NKW.Passkey.LastUsedDate': string;         // ISO 8601 UTC (or empty)
    readonly 'NKW.Passkey.BackupState': 'device-bound';  // reserved enum; only value in v1
    readonly 'NKW.Passkey.Transports': string;           // JSON array
}

// Protected field names that MUST be stored as ProtectedValue:
export const NKW_PASSKEY_PROTECTED_FIELDS: readonly string[] = [
    'NKW.Passkey.UserHandle',
    'NKW.Passkey.PrivateKey'
] as const;
```

**Migration policy:** `NKW.Passkey.Version === '1'` is the only accepted value for Phase 2. If core reads an entry with a different version, it must **ignore** the passkey fields (entry still opens normally) and log a warning. Adding version `'2'` later is allowed; dropping v1 support requires a migration flow.

**Empty / missing field semantics:**
- Any required field missing or empty → entry is treated as not-a-passkey. Core does not half-read.
- `LastUsedDate` may be empty string immediately after registration (never used yet).

## 3. Core-internal API (Feature A, already shipped — documented for Phase 3 reuse)

These interfaces are already implemented. Documented here so Phase 3 (#25, #39) reuses them consistently.

```ts
// packages/core/app/scripts/comp/passkey/passkey-prf.ts

export interface RegisterPasskeyOptions {
    rpId: string;                   // location.hostname
    rpName: string;
    userHandle: Uint8Array;         // 16 random bytes
    userName: string;
    userDisplayName: string;
    prfSalt: Uint8Array;            // caller-generated, 32 bytes
}

export interface RegisterPasskeyResult {
    credentialId: Uint8Array;
    prfOutput: Uint8Array | null;   // null if authenticator can't eval at registration
}

export function isPasskeyPrfSupported(): Promise<boolean>;
export function registerPasskey(opts: RegisterPasskeyOptions): Promise<RegisterPasskeyResult>;
export function evaluatePrf(credentialId: Uint8Array, prfSalt: Uint8Array): Promise<Uint8Array>;
export function wrapKey(wrapKey: Uint8Array, plaintext: Uint8Array): Promise<Uint8Array>;
export function unwrapKey(wrapKey: Uint8Array, ciphertext: Uint8Array): Promise<Uint8Array>;
```

```ts
// packages/core/app/scripts/comp/passkey/passkey-unlock.ts

// HKDF-SHA256 over PRF output, domain-separated by `info`.
// Do not add new callers without a new `info` string.
export type UnlockInfoTag =
    | 'neokeeweb-passkey-unlock-v1'    // Phase 2 Feature A
    | 'neokeeweb-passkey-perfield-v1'  // Phase 3 #25 (reserved)
    | 'neokeeweb-passkey-perurl-v1';   // Phase 3 #39 (reserved)

export function deriveUnlockKey(
    prfOutput: Uint8Array,
    salt: Uint8Array,               // fileId for unlock
    info: UnlockInfoTag
): Promise<Uint8Array>;              // 32 bytes
```

**Contract rule:** every reuse of the PRF credential MUST use a unique `info` string from the union above. Never reuse `neokeeweb-passkey-unlock-v1` for anything else or the two derived keys collide and unlock ciphertext can be decrypted by a per-field attack — classic key-separation bug.

```ts
// packages/core/app/scripts/models/file-info-model.ts (excerpt)

export interface FileInfoPasskeyFields {
    passkeyCredentialId: string | null;     // base64url

    passkeyWrappedKey: string | null;       // base64url, AES-GCM(wrap, masterPasswordUtf8)
    passkeyCreatedDate: string | null;      // ISO 8601 UTC
}

// All four are cleared atomically when passkey unlock is removed for a file.
```

## 4. Extension-internal API (Feature B, new)

```ts
// packages/extension/src/background/passkey/passkey-crypto.ts

export interface GeneratedPasskeyKeypair {
    credentialId: Uint8Array;        // 32 random bytes
    privateKeyPkcs8: Uint8Array;
    publicKeySpki: Uint8Array;
    coseAlg: -7 | -8;
}

export function generatePasskeyKeypair(alg: -7 | -8): Promise<GeneratedPasskeyKeypair>;

// Build the 37+ byte authenticator data blob per WebAuthn §6.1.
export interface AuthDataFlags {
    userPresent: boolean;            // UP
    userVerified: boolean;           // UV
    backupEligible: false;           // BE — always false for Phase 2
    backupState: false;              // BS — always false for Phase 2
    attestedCredentialIncluded: boolean; // AT — true on register, false on sign
}

export function encodeAuthenticatorData(
    rpIdHash: Uint8Array,            // SHA-256(rpId)
    flags: AuthDataFlags,
    signCount: number,               // uint32
    attestedCredentialData?: Uint8Array
): Uint8Array;

// SubtleCrypto returns IEEE P1363 (r||s); WebAuthn requires ASN.1 DER.
// This function is the single conversion point — test it hard.
export function p1363ToDerEcdsaSignature(p1363: Uint8Array): Uint8Array;

// CBOR-encode the attestation object with fmt='none'.
export function encodeAttestationObjectNone(authData: Uint8Array): Uint8Array;
```

## 5. Test fixtures and contracts

**Unit-test contract for `p1363ToDerEcdsaSignature`:** must round-trip against a known-good Web Crypto sign + verify cycle. Fixture: sign the same 32-byte hash 100 times with a fixed key; for each signature, convert to DER, parse with asn1.js, re-encode, assert byte-equality modulo DER canonicalization.

**Unit-test contract for `encodeAuthenticatorData`:** against WebAuthn spec test vectors (the spec has informative examples in §6.1 — encode them and byte-compare).

**Integration contract for the protocol verbs:** a contract test in `e2e/extension/passkey-protocol-contract.spec.ts` that:
1. Boots core with a synthetic vault.
2. Boots extension SW.
3. Dispatches each verb directly (skipping the shim) with fixture payloads.
4. Asserts on both the NaCl wire format (the encrypted envelope round-trips) **and** the decoded payload matching these types.
5. Asserts on KDBX entry state after `passkey-create-entry` / `passkey-sign`.

This contract test is the single source of truth for "does core and extension still agree". Breaking it = API bump + version change to `NKW.Passkey.Version`.

## 6. Change process for these interfaces

1. Propose a change as a comment on the sub-issue implementing that slice.
2. Update this file in the same PR as the implementation.
3. Do not merge an implementation that drifts from this file. If the implementation forced a type change, the doc change comes first.
4. `NKW.Passkey.Version` bumps any time the entry schema changes semantically (rename, remove, change meaning). New fields with sane defaults do not bump the version.

## 7. Out of band

- **NaCl session renegotiation:** unchanged from Phase 1. All 4 new verbs travel the existing encrypted channel. Stale-session handling (the 2026-04 extension-reconnect note in memory) applies unchanged — if the extension's session keys are stale after a core redeploy, the passkey verbs fail the same way autofill does, and the same reconnect flow applies.
- **Cross-package imports:** none. Core does not import from extension; extension does not import from core. Both import types from `packages/extension/src/background/protocol/types.ts` via workspace linking (core side adds a type-only import of that file). This mirrors the existing autofill protocol split.
